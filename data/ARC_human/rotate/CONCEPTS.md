# rotate 폴더 — Concept 설계

`rotate` 폴더의 concept은 그리드 안 객체를 **회전**시키는 변환을 다룬다. 파일 포맷/네이밍/색상 규칙은 루트 `HARNESS.md`를 따르고, 객체 분류·색상 구성·선택 근거 축은 `flip/CONCEPTS.md`와 동일한 설계를 재사용한다.

> 아래 설계는 대화 중 발견된 논리적 모순을 사용자와 확인 후 정리한 결과다. 원 아이디어와 달라진 부분은 각주로 표시.

## 0. 전제 — 객체 크기

**객체 크기는 항상 1×1 초과(다중 셀)로 제한한다.** `flip`과 동일한 이유: 1×1 객체는 회전해도 형태가 바뀌지 않아 rotate 고유의 의미가 없고, 위치만 바뀌는 경우는 `move`와 구분되지 않기 때문. *(원래 "1x1 이상"이었으나 1x1 제외로 확정)*

## 1. 기본 분류 (객체 개수) — flip과 동일

| 분류 | 설명 |
|---|---|
| **R-단일** | 그리드에 회전 대상 객체가 하나뿐 |
| **R-다** | 그리드에 여러 객체가 존재, 그중 하나만 회전 대상 |

### 1-1. R-단일: 객체 색상 구성 (3종)

1. **단색** — 객체를 이루는 모든 셀이 같은 색
2. **부분 다색** — 색상 종류가 1보다 많고 객체 크기(셀 수)보다는 적음
3. **전체 다색** — 색상 종류 수 = 객체 크기(셀 수)

*(원문의 "1 / 2 / 2" 번호는 "1 / 2 / 3" 오타로 간주하고 정리함)*

### 1-2. R-다: 객체 선택 근거 (3종)

1. **색상** 2. **크기** 3. **모양**

> R-다의 각 객체는 기본적으로 **단색(monochrome)** 이라고 가정한다 (flip과 동일한 이유).

## 2. 대상 객체 가로/세로 길이 (홀수/짝수)

- **1회전(90도, CW/CCW) + 이동 없음**: bbox가 정방형이어야 하므로(§3-1) 가로=세로, 값이 하나로 수렴한다.
- **그 외 모든 경우**(180도이거나, 이동이 있는 경우): bbox가 정방형이 아닐 수 있으므로 가로·세로 둘 다 관련되며, 각각 독립적으로 홀/짝을 지정한다 (같은 parity 값을 가로·세로 모두에 적용).

## 3. 회전 종류 (rotate 방향 × 횟수)

원래 "방향(cw/ccw) × 횟수(1회/2회)"의 4개 조합으로 제시되었으나, **(cw, 2회)와 (ccw, 2회)는 둘 다 180도 회전으로 시각적으로 동일한 변환**이라 사용자 확인 후 아래 **3종류**로 축소했다.

1. **90도 CW**
2. **90도 CCW**
3. **180도**

### 3-1. 정방형(bbox) 제약 — 이동 유무 및 회전 종류에 따라 다름

- **90도(CW/CCW) + 이동 없음**: bbox가 **정방형이어야 함** (필수) — 90도 회전은 h×w를 전치시키므로, 제자리(bbox 위치 불변)를 유지하려면 정방형이어야 하기 때문.
- **180도 + 이동 없음**: 180도는 치수(h,w)가 바뀌지 않으므로 **정방형 불필요**.
- **이동 있음**: 회전 종류와 무관하게 **정방형 불필요** (bbox가 어차피 새 위치로 옮겨가므로).

*(사용자 확인: 원래 "이동없음=정방형 필"이 회전 횟수 구분 없이 제시되었으나, 90도에만 적용하는 것으로 확정)*

## 4. 객체 회전 후 이동 유무

1. **이동 없음** — 객체는 회전(내부 픽셀 재배열)만 되고 bbox 위치는 원래와 동일
2. **이동 있음** — 회전과 별개로 bbox 위치 자체가 달라짐

### 4-1. 이동 있을 시: 기준점(pivot) 규칙

`flip`의 "기준선"과 달리 방향 축이 없다 — 회전은 이미 §3의 회전 종류(방향+각도)가 정해져 있으므로, 위치 이동도 **같은 회전 종류(방향/각도)로 기준점 주위를 도는 것**으로 정의한다. 즉 객체의 원래 앵커 좌표를 기준점을 중심으로 §3의 회전과 동일한 각도만큼 회전시켜 새 위치를 구한다.

**기준점 위치**
1. **bbox 내부** — 기준점이 객체 bbox와 겹침
2. **bbox 외부** — 기준점이 객체 bbox와 겹치지 않음

## 5. 조합 규모 — 전체 생성 완료

객체분류(R-단일 3종 + R-다 3종 = 6) × 홀짝(2) × 회전종류(3) × 이동유무(없음1 + 있음: 기준점위치2, 합3) = **6×2×3×3 = 108 조합 = 108커플 = 216 JSON 파일**. 전량 생성 완료 (§7 참고).

## 6. 파일 네이밍

HARNESS.md §9 규칙을 그대로 따른다: `rota000a`(2ex) / `rota000b`(3ex) 부터 시작해 필요시 `aa, ab, ...`로 확장.

## 7. 생성 결과 및 구현 노트

108개 조합 전체를 스크립트로 생성했다. 전체 매핑은 `rotate/MANIFEST.csv`에 있다 (idx, file_2ex, file_3ex, object_branch, sub, parity, rotate_type, move, pivot_pos 컬럼).

구현 시 확정한 세부 규칙 (`flip`과 공유하는 도형 생성·색상 구성·couple 색상 차이 로직은 동일, 차이점만 기술):

- **회전 변환**: §3의 90CW/90CCW/180 각각을 bbox 로컬 좌표에 대한 인덱스 변환으로 구현 (90도류는 h,w 치수가 서로 바뀜, 180은 안 바뀜).
- **기준점(pivot) 이동**: flip의 "기준선 반사"와 달리, 객체의 좌상단 앵커 좌표를 기준점 주위로 **§3과 동일한 회전(방향/각도)**을 적용해 새 위치를 구한다 — 내부 패턴 회전과 위치 이동이 같은 회전 파라미터를 공유하는 것이 flip과의 핵심 차이.
- **정방형 제약**: choose_hw에서 "90도+이동없음"일 때만 h==w를 강제하고, 그 외(180도, 또는 이동 있음)에는 h,w 각각 독립적으로 parity만 만족하면 되도록 구현.

**검증**: 216개 파일 전수 자동 검증(그리드 4~10, 배경0, 색 0-9, pair 내부 크기 고정, test 1개, 2ex=train2/3ex=train3, couple 색상 차이, no-op 없음, 컬러 픽셀 수 입출력 동일) 통과. 대표 샘플 6종(단일/mono/90CW 제자리, 단일/mono/180 제자리, 단일/mono/90CCW+pivot내부 이동, 다객체/shape, 다객체/size, 다객체/color)을 직접 grid로 계산해 대조, 전부 의도한 로직과 정확히 일치함을 확인.
