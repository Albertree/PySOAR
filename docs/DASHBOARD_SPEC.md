# dashboard 설계 명세

PySOAR(ARBOR) 풀이 과정을 키보드로 한 스텝씩 따라가며 보는 시각화 디버거다.
소스는 `arc/dashboard.py` 하나이고, 실행하면 자기 자신 안에 데이터를 끼워 넣은 단일 `arc/dashboard.html`을 만들어 낸다.
이 문서는 그 화면을 구성별로 나눠, 이 글만 보고도 다시 만들 수 있을 만큼 디테일하게 적어 둔 것이다.

실행: `python arc/dashboard.py <dataset> [limit]` → 같은 폴더에 `dashboard.html` 생성.


## 전체 구조

화면은 두 개다. 처음엔 문제 목록(task browser)이 뜨고, 문제를 고르면 풀이 화면(step viewer)으로 들어간다.
풀이 화면의 핵심 원칙은 **시스템 전체에서 원자적 변화 한 개가 일어날 때마다 그것을 한 스텝으로 본다**는 것이다.
WME 하나 추가, 규칙 하나 발화, operator 하나 제안 같은 단일 변화가 각각 별도의 스텝이고, 사용자는 이 스텝들을 좌우 방향키로 한 칸씩 넘기면서 매 스텝의 변화를 강조 표시로 확인한다.

스텝 데이터는 `arc/fine_trace.py`가 실제 PySOAR 사이클을 가장 잘게 쪼개 만든 트레이스에서 온다. 대시보드는 그 트레이스를 그리기만 한다.


## 페이지가 소비하는 데이터

`build()`가 `__DATA__` 자리에 JSON 한 덩어리를 박아 넣는다. 모양은 이렇다.

- `dataset`: 데이터셋 이름.
- `tasks[]`: 각 task마다
  - `id`: task id (예: `Teasy000a` 형태로 쓰되, 표시 네이밍 규약은 아래 WM 절 참고).
  - `events[]`: 스텝 배열. 각 이벤트는 `cycle`, `phase`, `kind`, `label`, `rule`(발화한 규칙명 또는 operator명), `highlight`(이 스텝에서 바뀐 WME 문자열 목록), `detail`(operator별 부가정보), `wm_state`(아래 wm_states 인덱스).
  - `wm_states[]`: WM 스냅샷들. 대부분의 스텝이 같은(큰) WM을 공유하므로, 서로 다른 상태만 한 번씩 저장하고 각 이벤트는 인덱스로 참조한다. 한 스냅샷은 `(id, attr, value)` 삼중쌍의 배열이다.
  - `grids`: `train`(input/output 쌍들), `test`(input만).
  - `candidates[]`: 답 후보. 각 후보는 `answer`(그리드), `position`, `color`.
  - `correct_attempt`: 정답을 맞힌 후보 인덱스(없으면 null).
  - `n_steps`.
- `rules[]`: production 매니페스트. 각 규칙은 `name`, `if[]`(조건: `id/attr/val/neg`), `then[]`(액션: `id/attr/val/pref`).
- `op_docs`: operator 이름 → 한 줄 설명.

핵심 파생값 두 개를 페이지가 직접 계산한다.

- **변경분(added/removed)**: 현재 스텝 WM과 *바로 이전 스텝* WM을 삼중쌍 문자열 집합으로 만들어 차집합. 추가된 줄과 삭제된 줄을 이렇게 얻는다.
- **조건 충족 여부**: 규칙 조건이 현재 WM 집합에 매칭되는지를 그 자리에서 판정(아래 rules 절).


## 색상 팔레트

CSS 변수로 한곳에 모은다. 어두운 테마.

- 배경 `--bg:#0f1117`, 패널 `--panel:#171a23`, 보조 패널 `--p2:#1d2130`, 경계선 `--line:#2a2f3e`.
- 본문 글자 `--txt:#d7dbe6`, 흐린 글자 `--muted:#8b93a4`.
- 강조 녹색 `--accent:#5fd97f`, 파랑 `--blue:#8ab4f8`, 금색 `--gold:#f3c969`.
- 추가 표시: 배경 `--add:#16361f` / 글자 `--addt:#7fe79a`.
- 삭제 표시: 배경 `--rm:#3a1820` / 글자 `--rmt:#ff8a8a`.
- 혼합(추가+삭제) 글자: `#ffb454`(주황).

ARC 그리드 색 팔레트(셀 값 0~9): `['#000','#0074D9','#FF4136','#2ECC40','#FFDC00','#AAA','#F012BE','#FF851B','#7FDBFF','#870C25']`.

기본 폰트는 13px 시스템 산세리프, 코드성 영역(WM·규칙·맵)은 등폭 폰트.


## 화면 1 — 문제 목록 (task browser)

위에 제목 `PySOAR dashboard — <dataset>`과 한 줄 안내(방향키로 선택, Enter로 풀이)가 있고, 그 아래 task 카드들이 그리드로 깔린다.

- 카드 그리드: `repeat(auto-fill, minmax(150px,1fr))`, 간격 12px.
- 카드 한 장: 패널색 배경, 2px 경계선, 둥근 모서리 10px. 위에 task id(등폭), 아래 메타(`N steps · solved (try k)` 또는 `unsolved`). 푼 문제는 녹색, 못 푼 문제는 빨강 글자.
- 선택된 카드: 경계선이 강조 녹색이고 같은 색 그림자 1px.
- 키: 좌우로 이웃 카드, 상하로 한 행 위/아래(현재 창 너비로 열 수를 계산해서 이동), Enter로 그 task 풀이 진입. 선택 카드는 항상 화면 안으로 스크롤.


## 화면 2 — 풀이 화면 (step viewer) 레이아웃

CSS Grid 한 판으로 전체를 잡는다. 화면 높이를 꽉 채우고(`height:100vh`), 안쪽 여백·칸 간격은 6px.

- 열: `300px | 1fr | 360px | 300px` (왼쪽부터 cycle map / working memory / rules / 오른쪽 묶음).
- 행: `auto | auto | 1fr | auto` (위부터 status bar / phase breadcrumb / 본문 / 하단 이벤트바).
- 상단 두 줄(status bar, phase breadcrumb)과 하단 이벤트바는 네 열 전체를 가로지른다(`grid-column:1/5`).
- 본문 행(`1fr`)에 네 패널이 한 열씩 들어간다.

모든 패널은 **고정 헤더 + 스크롤 본문** 구조다. 패널은 세로 flex이고, 헤더(`h3`)는 `flex:0 0 auto`로 고정, 본문은 `flex:1 1 auto; overflow:auto`로 스크롤된다. 헤더는 **불투명 배경 + 아래쪽 경계선**을 가져서, 본문을 스크롤해도 글자가 헤더 뒤로 비쳐 보이지 않는다. 이건 반드시 지켜야 한다.

각 패널 헤더 우측에는 작은 동그라미 `i` 아이콘이 있고, 마우스를 올리면 그 영역을 읽는 법과 색상 의미를 설명하는 툴팁이 뜬다. 화면 본문에는 해설성 문구를 넣지 않고, 설명은 전부 이 툴팁으로 보낸다.


### 상단 — status bar

현재 task id, `step k/N · cycle c`, 키 안내(좌우=스텝, Esc=목록, Home/End=처음/끝)를 한 줄에 두고, 맨 오른쪽에 이 task를 풀었는지(`solved (try k)` 녹색 / `unsolved` 빨강)를 붙인다.


### 상단 — phase breadcrumb

`input → propose → decide → apply → output` 다섯 단계를 알약 모양으로 늘어놓는다. 현재 스텝의 단계만 활성(파란 배경 `#2d4a86`, 녹색 경계선, 흰 글자)이고 나머지는 흐린 회색. 사이는 `→`로 잇는다.


### 왼쪽 — cycle map

이 task의 모든 이벤트를 위에서 아래로 한 줄씩 쌓은 목록이다. 폭이 넉넉해야 해서 첫 열(300px)을 통째로 쓴다.

- 각 줄은 등폭 11px. 현재 스텝 줄은 앞에 `▶`를 붙이고 파란 배경/흰 글자로 강조, 나머지는 앞에 공백.
- phase 경계 줄(kind가 phase인 줄)은 금색 굵은 글씨로 구분.
- 줄을 클릭하면 그 스텝으로 점프. 호버 시 옅은 배경.
- 한 줄이라도 가로폭을 넘으면 줄바꿈하지 않고(`white-space:nowrap`) 가로 스크롤로 본다.
- 현재 스텝 줄은 항상 보이도록 스크롤.
- 세로 목록이라 방향키 상하로도 이전/다음 스텝 이동이 된다(아래=다음, 위=이전).


### 가운데 왼쪽 — working memory

S1을 루트로 한 단일 트리 한 개다. 자세한 규칙은 아래 "WM 표현" 절에 따로 모았다. 헤더 우측에 전부 열기/닫기 버튼(`⇕ all`)이 있다.


### 가운데 — rules

에이전트의 기본 production 규칙들을 카드로 나열한다. 자세한 규칙은 아래 "rules 표현" 절. 헤더 우측에 `⇕ all` 버튼.


### 오른쪽 — problem + candidates

오른쪽 열은 세로로 두 패널을 쌓는다(세로 flex, 간격 6px).

- 위 `problem`(최대 높이 42%): train 예제쌍과 test 입력을 보여준다. 한 쌍은 가로 한 줄에 `캡션 input → output` 형태로 둔다. 예제쌍들은 세로로 쌓이고, 각 쌍 내부의 input→output은 가로로 흐른다. test는 `test_k input → ?` 로 출력 자리를 물음표로 둔다. 그리드는 작은 10px 셀 테이블로 그린다.
- 아래 `answer candidates`(남은 높이): 답 후보들. **WM에 `(S1 ^answer-ready ...)`가 올라온 뒤부터만** 후보를 그린다(그 전엔 비워 둔다). 후보 카드는 캡션 `cand{i} ✓`(정답이면 체크), 그리드, 그 아래 흐린 글씨로 position/color. 후보 위에는 현재 operator의 부가정보(아래 detail) 블록을 같이 보여준다.


### 하단 — 이벤트 디테일 바

네 열을 가로지르는 낮은 바(최대 96px, 넘으면 스크롤). 현재 스텝이 무엇을 했는지를 보여준다.

- 큰 글씨로 이벤트 라벨(최대 120자, 넘으면 `…`). 라벨 색은 이벤트 종류별로 다르다: 규칙 발화·WME 추가는 녹색, 규칙 철회·WME 삭제는 빨강, operator 제안/선택은 금색, decide는 파랑, output은 강조 녹색, no-op은 회색.
- 그 아래 흐린 글씨로 `phase=... · kind=... · rule/op=... · changed: ...`. changed에는 바뀐 WME를 최대 4개까지 보여주고 더 있으면 `(+n more)`로 줄인다. 각 항목은 40자에서 `…`로 자른다.

operator별 부가정보(detail)는 오른쪽 candidates 위에 그린다.

- observe: 각 pair의 객체 color와 cells.
- compare: pair별 입력좌표/색 → 출력좌표/색.
- generalize: 인자별로 resolve된 일반 표현식 목록과 `out=make_grid(size,fill)+coloring(pos,color)` 요약.
- compose: 조립된 답 그리드.


## WM 표현

WM은 **S1을 루트로 한 트리 한 개**이고, 보이는 모든 줄은 예외 없이 `(id ^attr value)` 삼중쌍이다. input-link, output-link도 별개 메모리가 아니라 이 한 트리 안의 가지(`^io.input-link`, `^io.output-link`)다. 맨 앞에 `S1 (top state)` 같은 래퍼 줄은 두지 않고 곧장 WME들을 펼친다(WM은 순서 없는 집합이라 "맨 위" 개념이 없다 — 줄 순서는 표시용일 뿐).

### 트리 만드는 법

WM 삼중쌍들을 노드 그래프로 모은다. 어떤 삼중쌍의 value가 다른 노드의 id이면 그건 자식 가지(edge), 아니면 그 노드의 속성(property)으로 친다. 그래서 task→pair→grid→object 같은 ARCKG 계층이 자연히 토글 트리로 펼쳐진다.

### 토글과 lazy

- 가지는 `▸/▾` 토글(`<details>`)로 펼친다. **기본은 닫힘.**
- 자식 노드에 아직 내용(속성·자식)이 없으면 토글을 만들지 않고 `(... ) (lazy)` 한 줄로 둔다. 즉 Task가 막 들어온 시점엔 pair/grid 같은 하위가 lazy 상태로만 보이고, observe 등으로 방문(visit)해야 속이 채워진다.
- **토글 열림 상태는 스텝 사이에 유지된다.** 노드 id가 트레이스 내내 안정적이므로, 어떤 토글을 열어 두고 다음 스텝으로 넘어가면 계속 열려 있다. 사용자가 열고 닫은 것을 노드 id 집합(`wmOpen`)으로 기억한다.
- 다음 스텝에서 *새로 생긴* 토글은 그 집합에 없으므로 닫힌 채로 시작한다.

### 변경 마커와 색

각 줄 맨 앞에 고정폭(1em, 가운데 정렬) 마커 칸을 둔다. 이 칸이 고정폭이라 토글을 여닫거나 스텝을 넘겨도 줄이 좌우로 흔들리지 않는다. 마커는 네 가지.

- `+` 그 줄의 WME가 이번 스텝에 추가됨 → 줄 전체를 추가색(녹색 배경+녹색 글자).
- `−` 그 줄의 WME가 삭제됨 → 줄 전체를 삭제색(빨강 배경+빨강 글자).
- `~` 그 줄 자신은 그대로지만 *하위 어딘가*가 바뀜 → 배경은 칠하지 않고 **글자색만** 바꾼다(추가=녹색, 삭제=빨강, 추가·삭제 섞이면 주황).
- 공백 변화 없음.

여기서 핵심은, 하위에서만 바뀐 상위 토글은 절대 배경을 칠하지 않고 글자색만 바꾼다는 점이다. 예를 들어 어떤 grid 노드가 추가되면 그 위의 pair, task 토글까지 줄줄이 글자만 녹색이 된다(`~`). 변경 상태는 바뀐 WME의 id에서 부모를 타고 위로 끝까지 전파해서 칠한다.

### 값 생략

속성값이 길면 40자에서 `…`로 자른다. 특히 `grid.contents` 같은 큰 값은 이미지로 그리지 않고 잘린 문자열로만 보여준다(그리드 그림은 오른쪽 problem/candidates 영역에서만 그린다). dict/list 값은 JSON으로 직렬화한 뒤 자른다.

### operator의 WM 표현

operator가 제안되면 추상 id로 객체가 만들어진다. 이름은 `O1, O2, ...` 식으로 짓고, 그 객체가 무슨 operator인지는 별도 WME `(O1 ^name observe [^arg ...])`로 명시한다. operator 이름을 id에 붙여 `Oobserve`처럼 만들지 않는다.

- propose 단계에서 acceptable preference `(S1 ^operator O1 +)`가 WM에 올라오고, 그 operator가 무엇인지 알 수 있도록 `(O1 ^name observe ...)` 정의가 같이/뒤따라 들어온다. candidate가 WM에 추가되는 스텝에는 실제로 WM 변화가 동반되어야 한다.
- decide에서 하나가 선택되면 나머지 후보는 사라진다. 우리 구현은 사이클당 후보가 하나라, 그 후보의 acceptable preference는 proposal이 철회되는 시점(apply에서 결과 flag가 set되어 propose 규칙의 조건이 깨질 때)에 사라진다.


## rules 표현

`rules` 매니페스트의 모든 규칙을 카드로 나열한다. 각 카드는 접을 수 있고, 기본은 **닫힘**(WM 토글과 같은 정책).

- 카드 제목은 규칙명(금색 굵게). 이번 스텝에 발화한 규칙이면 제목 옆에 `● fired`를 붙이고, 카드 테두리를 강조 녹색으로 두른다. 접혀 있어도 테두리로 발화를 알 수 있고, 테두리는 접힌 크기/펼친 크기에 맞춰 생긴다.
- 카드를 펼치면 `IF` 라벨 한 줄, 그 아래 조건들이 각자 한 줄씩, 다음에 `THEN` 라벨 한 줄, 그 아래 액션들이 각자 한 줄씩 나온다. 조건마다 자기 줄을 차지하고 줄바꿈하지 않는다.
- 열림 상태는 규칙명 집합(`ruleOpen`)으로 스텝 사이에 유지한다. 다음 스텝에서 새로 생긴 규칙은 닫힌 채로 시작.

### 조건 색

조건은 종류에 따라 아주 옅은 배경색을 깐다.

- 긍정 조건(WM에 있어야 함): 옅은 파랑 배경 `rgba(138,180,248,.11)`.
- 부정 조건(WM에 없어야 함): 옅은 빨강 배경 `rgba(255,138,138,.12)`. 부정 조건은 패턴 앞에 `-` 기호를 붙여 표시한다(SOAR 표준의 부정 prefix). 이 `-` 기호는 조건 색을 그대로 따른다.

글자색은 기본 회색(미충족)이고, 그 조건이 현재 WM에서 충족되면 배경색 계열의 진한 글자(긍정=진한 파랑, 부정=진한 빨강)로 바뀐다. 충족 강조에 녹색은 쓰지 않는다(녹색은 발화한 규칙 테두리 전용).

조건 충족 판정은 이렇게 한다. value가 변수(`<x>` 꼴)면 `(id ^attr ` 로 시작하는 WME가 하나라도 있는지로, 아니면 정확한 삼중쌍이 WM 집합에 있는지로 본다. 부정 조건이면 그 결과를 뒤집는다. 한 규칙은 모든 조건이 충족(=진한색)일 때만 발화한다.

### THEN 표기

액션 한 줄은 `(id ^attr` 까지는 본문 회색, 제안되는 값(operator 등 key WME 값)은 금색, 마지막 preference 기호와 닫는 괄호는 회색으로 둔다. then 전체를 한 색(노란색)으로 칠하지 않는다.

### operator apply 카드

WM을 바꾼 주체가 production 규칙이 아니라 operator body인 스텝(현재 이벤트의 `rule`이 매니페스트에 없는 경우)에는, 규칙 목록 위에 `operator <name> apply` 카드를 따로 얹고 그 아래 `op_docs`의 한 줄 설명을 단다.


## 전부 열기 / 닫기 버튼

WM·rules 패널 헤더 우측의 `⇕ all` 하나로 그 패널의 모든 토글을 제어한다. 동작은 단일 토글이다.

- 지금 보이는 토글 중 **하나라도 닫혀 있으면 전부 연다.** 전부 열려 있으면 전부 닫는다.
- "전부 열기"는 그 시점에 화면에 있는 토글의 id만 열림 집합에 넣는다. 그래서 다음 스텝에서 새로 생기는 토글은 여전히 닫힌 채로 시작한다.


## 키 조작

- 문제 목록: 좌우=이웃 카드, 상하=한 행 위/아래, Enter=풀이 진입.
- 풀이 화면: 오른쪽/아래=다음 스텝, 왼쪽/위=이전 스텝(cycle map이 세로라 상하도 매핑), Home=첫 스텝, End=마지막 스텝, Esc=문제 목록으로.


## 사이클 흐름에서 화면이 지켜야 할 점

- 한 사이클은 input에서 시작해 output으로 끝나고, output 단계에서 output-link 내용을 (있든 없든) 내보낸다. 화면은 output에 무엇이 있는지/없는지를 보여줘야 한다.
- 정적 문제는 다음 사이클의 input이 사실상 no-op이지만, 그래도 두 번째 이후 사이클의 input 단계를 생략하지 말고 한 스텝으로 보여준다.
- test가 여러 개여도 한 번에 한 문제만 제출한다(동시 제출하지 않는다). 후보 제출은 랭킹 순서대로 내고 틀리면 다음 후보로, 최대 3회.
